How it works

Attribution — which component is responsible?

Paired Δ on success vs baseline. Red = the 95% CI excludes zero (a real effect, not noise).

Tool value (Shapley credit)

Fair credit for each tool toward success, averaged over coalitions.

Sensitivity — success vs temperature

Higher sampling temperature trades reliability.

A/B verdict — strong vs weak model

Wilson CIs + two-proportion z-test + effect size (borrowed from MultiEval).

Counterfactual (but-for) tool usefulness

Leave-one-out: remove exactly one tool, hold the rest fixed. The causal question you ask when pruning an agent. Complements Shapley.

Propagation through the chain — Q1–Q4

Change one component in one agent; measure the paired effect on the final answer and on downstream stages. Red = significant.

Pareto explorer — accuracy vs latency vs cost

The one idea

Every agent run — response = f(prompt, model, tools | query) — is reduced to one RunTrajectory. All evaluation runs on that object, so the simulator and a real model are interchangeable, and the same attribution code works for prompt, model, tools, and every config lever.

Trustworthy, not noisy

Seeded repeats, bootstrap CIs, paired differencing, and an effect is only "real" if its interval excludes zero. The whole run is reproducible across processes (a determinism bug was found and fixed + regression-tested).

Trustworthy enough to gate what ships

An evals-as-CI gate blocks a release unless the confidence-interval lower bound clears a threshold — statistical confidence, not a lucky average. 54 tests + the gate run green on every push.

Self-validating

The default subject is a simulator with planted ground truth (each model's capability, which tools a task needs). A good evaluator must recover it — and it does (weak models and vague prompts show significant drops; the calculator tops the tool ranking). Only then is it pointed at real models.
